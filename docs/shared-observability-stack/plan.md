# Plan: Shared Observability Stack

## Approach
Stand up one Compose stack — Prometheus, Grafana, Loki, Grafana Alloy, node_exporter, cAdvisor — at `compose/desktop/observability/`, deployed once under a dedicated `observability` nologin service user, reusing `scraper-egress`'s service-user/sudoers/`.env`-seeding skeleton (ADR 0014 precedent) with a directory-recursive rsync step in place of scraper-egress's flat-file copy (see Integration points — the two deploy.sh scripts are not structurally identical). Prometheus scrapes node_exporter and cAdvisor **directly** via native `scrape_configs` — this is the standard Prometheus pull model and the only way its `/targets` page can show them as `UP` (AC5). Alloy is used **only** for the Docker log-discovery-and-shipping path: `discovery.docker` enumerates running containers, `loki.source.docker` tails their logs, `loki.write` ships them to Loki. Alloy plays no role in metrics collection — there is no `remote_write` receiver on Prometheus and no unauthenticated write endpoint. Grafana's datasources and dashboards are provisioned entirely as code (files under `dashboards/` and `provisioning/`), so a fresh deploy needs zero manual UI steps. The two legacy Grafana instances and financial-pipeline's Loki keep running unmodified throughout — this is additive infrastructure, not a migration, until Preston separately repoints/decommissions the old stacks.

Data flow summary:
1. node_exporter and cAdvisor expose `/metrics` endpoints on `observability_net`; Prometheus scrapes both directly via `scrape_configs` `static_configs` jobs — no intermediary. This is what makes AC5 ("targets page shows node_exporter and cAdvisor as UP") true: they are real Prometheus scrape targets, not remote-write series.
2. Alloy's `discovery.docker` component enumerates running containers on the desktop's Docker socket; `loki.source.docker` tails each container's log stream, a relabel/process stage keeps only stable labels (container name minus the compose-hash suffix, compose project/service) and drops churning ones (full container ID, ephemeral labels), then `loki.write` forwards to Loki's push API. Alloy has no `prometheus.scrape` or `prometheus.remote_write` component — it is not part of the metrics path at all.
3. Grafana queries Prometheus (PromQL) and Loki (LogQL) through provisioned datasources with fixed UIDs (`uid: prometheus`, `uid: loki`); dashboards are pre-loaded JSON referencing those literal UID strings directly (not `${DS_PROMETHEUS}`/`${DS_LOKI}` template variables — those only resolve through Grafana's UI import wizard, not file-based provisioning).
4. Grafana is bound to the desktop's LAN interface only (Internal Service, no Cloudflare Tunnel, per requirements out-of-scope #7). Prometheus and Loki are also host-published (see API/interface contract) for the documented consumer-attach contract, with no auth in front of either — an accepted LAN-only tradeoff (see Risk areas).

## Data model
No data model changes. This is infrastructure, not application state. Prometheus, Grafana, and Loki persist to **bind mounts** under `/opt/docker/observability/{prometheus,grafana,loki}-data`, chowned to the `observability` UID/GID by `deploy.sh` — not named Docker volumes (see Technology choices for why bind mounts were chosen over the original named-volume design). Alloy also gets a bind-mounted `alloy-data` directory to persist its own position/state tracking (used by `loki.source.docker` to avoid re-shipping or losing its log-tailing cursor across restarts) — the original plan had no persistence for this at all.

## API / interface contract

**Ports (desktop):**
- Grafana: `3300`, host-published, LAN-reachable (documented in README; distinct from 3001/3200 per FR11; configurable via `.env` `GRAFANA_PORT`).
- Prometheus: `9090`, host-published (standard default, not currently in use on desktop). Host-published deliberately — the Consumer Attach Contract below requires future consumer repos to reach Prometheus at `10.0.0.243:9090`. No auth in front of it; see Risk areas.
- Loki: `3100`, host-published. This port is free on the desktop today — `financial-pipeline-loki-1` currently has **no** host-published port at all (verified), so there is no actual collision risk; `3100` is simply Loki's own standard default port, chosen for that reason, not because it "matches" anything. Host-published deliberately for the same consumer-attach reason as Prometheus; no auth in front of it; see Risk areas.
- Alloy UI: `12345` — bound to **loopback only** (`127.0.0.1:12345:12345`), never the LAN interface. Useful for verifying pipeline health from the desktop itself (`ssh desktop-agent` + local curl, or an SSH tunnel), not exposed to the LAN.
- node_exporter: `9100`, cAdvisor: `8080` — **no `ports:` block in docker-compose.yml, ever.** Reachable only via the shared `observability_net` bridge network, for Prometheus to scrape. This is a hard requirement, not a style preference: host port `8080` is already bound by `gluetun` (arr-stack) on this desktop (verified via SSH) — publishing cAdvisor on `8080:8080`, as its tutorial-default config commonly shows, would fail the deploy outright.

**CLI / operational surface:**
- `deploy.sh` (Mac-run, reuses scraper-egress's service-user/sudoers/`.env`-seeding skeleton): creates the `observability` service user + directories on desktop, captures the real `id -u observability` / `id -g observability` immediately after `useradd --system` and writes those values into the deployed `.env` as `PUID`/`PGID` (the plan does not assume `999:999` — AC1's UID/GID check must read whatever `deploy.sh` actually wrote), installs `/etc/sudoers.d/observability`, chowns the bind-mount data directories to that UID/GID, does a directory-recursive rsync of the compose file plus `alloy/`, `prometheus/`, `loki/`, `grafana/provisioning/`, and `dashboards/` (unlike scraper-egress's flat 3-file rsync — see Integration points), generates a random `GRAFANA_ADMIN_PASSWORD` if `.env` doesn't already have one set (never leaves Grafana at its `admin:admin` default), seeds `.env` from `.env.example` if absent (never overwrites a populated `.env`), and `chmod 0600`s the deployed `.env`. `deploy.sh` never echoes or logs `.env` contents and never runs `set -x` around any command that touches `.env` or secret values.
- On desktop, day-to-day operation: `sudo -u observability docker compose -f /opt/docker/observability/docker-compose.yml up -d / down / ps / logs`, gated by the sudoers entry (never raw `docker` for that user).
- Consumer attach contract (documented in README, per FR13): logs are push-based — a consumer's own Alloy/Promtail instance points `loki.write` at `10.0.0.243:3100`'s push API instead of running its own Loki. Metrics are pull-based — because there is no remote-write receiver on this Prometheus (removed, see Approach/Architecture), a consumer's own app-level `/metrics` endpoint is attached by adding a new `static_configs` target to **this stack's** `prometheus.yml`, a small follow-on edit made when a consumer actually attaches, not by the consumer pushing to `10.0.0.243:9090`. `10.0.0.243:9090` being host-published is for querying Prometheus's HTTP/PromQL API directly (e.g. a consumer's own Grafana or tooling), not for writing to it. This is documentation only in this phase — no consumer repo is edited (FR15/out of scope).

**Errors / failure modes surfaced to the operator:**
- Prometheus targets page (`:9090/targets`) shows `DOWN` with scrape error text if node_exporter/cAdvisor are unreachable — this is now a real, populated targets page (AC5), not an empty one.
- Alloy UI (`:12345`, loopback-only) shows component health/graph and per-component error state (e.g. `loki.source.docker` failing to reach the Docker socket).
- Grafana datasource test (provisioned, but "Test" button in UI still works) surfaces connection errors to Prometheus/Loki.
- All six containers define `healthcheck:` blocks in docker-compose.yml; `docker compose ps` surfaces `healthy`/`unhealthy` status directly (AC4's "running and healthy" check depends on this).

## Technology choices
- **Grafana Alloy** (not Grafana Agent, deprecated) — mandated by requirements, scoped narrowly here to what it's actually needed for: Docker log discovery and shipping (`discovery.docker` + `loki.source.docker` + `loki.write`). It is not used for metrics collection.
- **Prometheus native `scrape_configs`** for node_exporter and cAdvisor, in place of Alloy `prometheus.scrape` + `prometheus.remote_write` to a Prometheus remote-write receiver. This is the standard Prometheus pull model, it's what makes the `/targets` page populate correctly (AC5 is architecturally impossible under the remote-write design), and it removes an unauthenticated write endpoint (`--web.enable-remote-write-receiver` accepts pushes from any client on `observability_net`/the LAN with no auth — a cardinality-bomb/DoS vector) that the original design never flagged as a risk. The tradeoff — scrape targets now live in `prometheus.yml` instead of being centralized in Alloy's config — is accepted; it's the conventional place for them to live anyway.
- **cAdvisor** (`gcr.io/cadvisor/cadvisor`) — the standard per-container metrics exporter that reads cgroups + the Docker socket; needed to satisfy FR4(b)/FR5 (per-container CPU/mem/network/disk) since Alloy itself doesn't collect container resource metrics, only logs. Runs as its image default (root-equivalent inside its own container) — see Risk areas.
- **node_exporter** (`prom/node-exporter`) — standard host-metrics exporter for CPU/mem/disk/network (FR4a/FR5); requires host-path mounts (`/proc`, `/sys`, `/`) read-only, standard for this exporter; runs pinned to the `observability` UID like the other non-socket services.
- **Grafana file-based provisioning** (`/etc/grafana/provisioning/{datasources,dashboards}`) — Grafana's built-in as-code mechanism, avoids any custom tooling and satisfies FR8/FR10 directly with no additional dependency. Datasource UIDs are pinned explicitly (`uid: prometheus`, `uid: loki`) rather than left auto-generated, so dashboard JSON can reference stable literal UIDs.
- **Bind mounts** (`/opt/docker/observability/{prometheus,grafana,loki,alloy}-data`, chowned to `observability` by `deploy.sh`) rather than named Docker volumes. This matches actual repo precedent — both `scraper-egress` and `embed-stack` use bind mounts, not named volumes — and is directly scriptable for the repo's existing NAS-backup pattern (streaming host filesystem paths over SSH). It also doesn't pretend to solve a UID problem that named volumes never solved either way: Prometheus/Grafana/Loki images ship their own baked-in default UIDs (Prometheus `65534`/nobody, Grafana `472`, Loki `10001`), which would conflict with a single custom `observability` UID under named-volume default ownership exactly as much as under a bind mount. The actual reconciliation is the explicit `user:` override on each service plus `deploy.sh` chowning the bind-mount directory to that same UID/GID beforehand — the image's baked-in default UID becomes irrelevant once the container is told to run as a different one and the mounted directory already has matching permissions.
- **Image tag pinning across all six services** (Grafana, Prometheus, Loki, node_exporter, cAdvisor, Alloy) — specific version tags, never `:latest`, to avoid silent drift; especially important here since imported dashboards assume specific Grafana schema versions.

## Risk areas
- **cAdvisor and Alloy both need `/var/run/docker.sock`, which is incompatible with the `observability`-UID `user:` pin used elsewhere in this stack.** `/var/run/docker.sock` is owned `root:docker` mode `660` on a standard Docker host; a non-root container process outside the `docker` group cannot open it, and cAdvisor's image additionally expects to run as root to read cgroups. Pinning `cadvisor` and `alloy` to the non-root `observability` UID would crash-loop both — the two services most central to this stack's metrics-target-discovery and log-shipping jobs. **Resolution:** `cadvisor` and `alloy` are explicitly carved out of the `user:` pin (see Architecture, Integration points) and run as their image defaults. This is **not** a violation of CONVENTIONS.md section 1 or requirements FR6/FR7: those rules govern the privilege boundary of the **host `observability` service user** — it never joins the `docker` group, never runs as root itself, and its only Docker access is the narrow, command-scoped `/etc/sudoers.d/observability` grant (same shape as scraper-egress's). They do not, and cannot, constrain what UID a container's *internal* process runs as when that process's documented job (cAdvisor reading cgroups + the socket, Alloy's `discovery.docker`) inherently requires root/socket access inside its own container. The socket mount is still read-only (`:ro`) where the image supports it. Document this reconciliation clearly in the README so it isn't misread as a FR7 violation during review.
- **Prometheus and Loki are unauthenticated APIs, host-published to the LAN.** Both need to be host-published (not just reachable inside `observability_net`) to satisfy the documented Consumer Attach Contract (FR13) — even though no consumer actually attaches in this phase. Neither Prometheus's HTTP/PromQL API nor Loki's push/query API has any auth in front of it. This is a documented, accepted tradeoff: LAN-only reachability, single-operator household, no external exposure (no Cloudflare Tunnel, per out-of-scope #7). Full auth (Authentik/OIDC in front of Grafana/Prometheus/Loki) was considered and is explicitly deferred — single-operator + LAN-only is judged sufficient for now; revisit via a follow-up ADR if this stack is ever exposed beyond the LAN or a second operator is added.
- **Unbounded disk growth from metrics/log retention.** Requirements scope out a long-term retention *policy*, which is different from having no default ceiling at all — Loki has no built-in retention unless configured, and Prometheus's default retention is 15 days but unbounded by size. The desktop also hosts Plex media storage, so this is a real disk-fill risk, not a long-tail edge case, especially once FR13 consumers attach later and add volume. **Resolution:** conservative baseline retention is set as infra hygiene, independent of "policy" being out of scope — Prometheus `--storage.tsdb.retention.time=15d` plus a size cap `--storage.tsdb.retention.size=5GB` (adjustable), Loki `limits_config`/compactor `retention_period: 336h` (14d). See `prometheus.yml` and the new `loki/loki-config.yml` in Integration points.
- **Cardinality footguns in the Loki log pipeline and the cAdvisor scrape.** Unfiltered `loki.source.docker` attaches high-cardinality/churning labels (full container ID, compose-hash container name) as Loki stream labels — Loki's most common operational footgun, and one that compounds with the retention risk above. Unfiltered cAdvisor scrape emits very high-cardinality per-container/per-device metrics on a host running 50+ containers. **Resolution:** Alloy's log pipeline gets a relabel/process stage keeping only stable labels and dropping churning ones; the Prometheus `cadvisor` scrape job gets `metric_relabel_configs` (or cAdvisor's `--disable_metrics` flag) to drop unneeded metric families. See `alloy/config.alloy` and `prometheus.yml` in Integration points.
- **Dashboard JSON re-pointing, de-dup, and alert-rule stripping (FR9).** Exported dashboards from `fashion-monitor-grafana-1`/`financial-pipeline-grafana-1` embed datasource UIDs specific to those old instances and may embed alert-rule/contact-point definitions (no Alertmanager exists in this stack) and could collide on dashboard `uid` if both source instances were built from a common starter template. All three need a mechanical fix-up pass before drop-in (re-point panel `datasource` fields to the literal `prometheus`/`loki` UIDs, de-dup/rename top-level `uid` fields, strip embedded alert-rule blocks) — easy to miss per-panel in a large dashboard, so this needs an actual verification pass (AC7: "renders successfully... with data"), not just file-presence.
- **Resource ceiling is undefined.** Requirements flag "[threshold TBD]" for CPU/memory at idle — six new containers (cAdvisor and Prometheus tend to be the heavier ones) alongside Plex/gaming GPU load and the ollama broker's CPU inference lane could contend. Recommend setting conservative `mem_limit`/`cpus` on each service (following the embed-stack/scraper-egress precedent of explicit caps) and spot-checking `docker stats` post-deploy, but there's no numeric target to validate against — flag this back to Preston rather than inventing a number.
